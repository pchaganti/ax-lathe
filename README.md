# Lathe

Generate hands-on technical tutorials on demand, then read them in your browser with verified working code.

Lathe targets topics where good documentation is scarce — *"build a digital synth in Zig"*, *"build a 3D slicer from scratch"*, *"build a database from scratch in Go"*. You get an expert guide that walks you through the work step by step, with each checkpoint independently runnable. Tutorials are generated by a Claude Code skill, then stored, served, and verified by a Go CLI.

## How it works

Two layers with a clean boundary:

- **Claude Code skills** — generate and work with tutorials, all run in your **interactive** Claude Code session: `/lathe` writes `part-01.md`, `/lathe-extend` adds the next part, `/lathe-verify` works through a tutorial to confirm it compiles and runs, and `/lathe-ask` answers questions about a part you're reading. Running them interactively keeps the work on your Claude subscription (headless `claude -p` is metered as of 2026-06-15; interactive use is not).
- **`lathe` CLI** (Go) — copies tutorials into `~/.lathe/tutorials/`, serves the rendered output at `http://localhost:4242`, and owns all durable state. It never calls `claude` itself: the web buttons and the `lathe verify`/`lathe extend` commands just hand you the skill command to paste into your session, and the skills call back into the CLI (`lathe store`, `lathe verify-result`, `lathe extend-start`/`extend-commit`) to record results.

```
User: /lathe "build a digital synth in Zig"     (interactive Claude Code)
        │
        ▼
  [/lathe skill]                     generates part-01.md
        │
        ▼  lathe store /tmp/lathe-<slug>
  [lathe CLI]                        copies files (status: unverified)
        │
        ├── ~/.lathe/tutorials/<slug>/
        │       metadata.json        status: unverified
        │       part-01.md, part-02.md, …
        │
        └── lathe serve → "Verify"/"Add Part"/"Ask" buttons hand you a command
                │
                ▼  paste into your session: /lathe-verify <slug>
            [/lathe-verify skill, interactively]
                works through each step, runs every checkpoint command, then
                `lathe verify-result` updates metadata: verifying → verified | failed | skipped

User: lathe serve  →  http://localhost:4242  (browser opens automatically)
```

## Install

Requires Go 1.25+ and the `claude` CLI on `$PATH`.

```bash
go install github.com/devenjarvis/lathe@latest
```

Or build from source:

```bash
git clone https://github.com/devenjarvis/lathe
cd lathe
go build -o lathe
```

The skills live under `.claude/skills/` in this repo — `lathe/`, `lathe-verify/`, `lathe-extend/`, and `lathe-ask/`, each a `SKILL.md`. Copy them into your own project's `.claude/skills/` (or your user-level skills directory) so Claude Code can discover them.

## Usage

Generate a tutorial inside any Claude Code session:

```
/lathe build a digital synth in Zig
```

Then open it:

```bash
lathe serve              # starts the web server, opens the browser
```

Other commands:

```bash
lathe list               # show all stored tutorials with status badges
lathe open <slug>        # open a specific tutorial (requires lathe serve)
lathe store <path>       # save a tutorial directory manually (status: unverified)
lathe store <path> --verify   # save, then print the /lathe-verify command to run
lathe verify <slug>      # print the /lathe-verify <slug> command to run in your session
lathe extend <slug>      # print the /lathe-extend <slug> command to run in your session
lathe rm <slug>          # delete a stored tutorial (prompts unless --force)
```

You can also delete tutorials from the web UI — each row on the list page has a `×` button that removes the tutorial after a confirmation.

Default port is `4242`; override with `--port`.

## Storage layout

Tutorials live globally in `~/.lathe/tutorials/`, one directory per slug:

```
~/.lathe/tutorials/
  digital-synth-zig/
    metadata.json
    part-01.md
    part-02.md
    part-03.md
  database-from-scratch-go/
    metadata.json
    index.md
```

`metadata.json`:

```json
{
  "slug": "digital-synth-zig",
  "title": "Build a Digital Synth in Zig",
  "topic": "build a digital synth in Zig",
  "created": "2026-05-03T19:00:00Z",
  "status": "unverified",
  "series": true,
  "parts": ["part-01.md", "part-02.md", "part-03.md"]
}
```

Status is one of `unverified` (the default after `lathe store`; renders no badge), `verifying`, `verified`, `failed`, or `skipped`. On failure, a `verify-result.json` is written alongside with the failed part, step number, and error output; the web UI renders it as a panel on the tutorial page.

## Verification

Verification is **opt-in** and runs in your interactive Claude Code session. Storing a tutorial leaves it `unverified` — nothing runs until you ask. The `lathe verify <slug>` command, the `--verify` flag on `lathe store`, and the **Verify this tutorial** button in the web UI all just hand you the same command to paste into your session:

```
/lathe-verify <slug>
```

The `/lathe-verify` skill works through every step in the tutorial — creating files in a fresh `mktemp -d` scratch dir (never your repo), running commands, executing each `## Checkpoint` block — then calls `lathe verify-result` to record the outcome in the tutorial's `metadata.json`. It marks the run `verifying` when it starts and a terminal `verified` / `failed` / `skipped` when it finishes; while a run is in flight the web page auto-refreshes every 5s, so the badge moves ⏳ → ✅ / ❌ / ⚠️ on its own.

Verification only makes sense where the tutorial's toolchain is installed. If a required tool is missing (e.g. no `zig` binary), the run is reported as **skipped** (⚠️) rather than failed — "couldn't verify here" is not the same as "broken."

Because verification now runs in your own interactive session, it executes under your normal Claude Code permission model — you see and approve the tool calls. The scratch-dir convention keeps build artifacts out of your repo, but treat it as soft isolation, not a security boundary.

## Dependencies

- [`spf13/cobra`](https://github.com/spf13/cobra) — CLI command structure
- [`yuin/goldmark`](https://github.com/yuin/goldmark) + [`goldmark-highlighting`](https://github.com/yuin/goldmark-highlighting) — markdown rendering with Chroma syntax highlighting
- `claude` CLI — Claude Code, where the `/lathe`, `/lathe-verify`, `/lathe-extend`, and `/lathe-ask` skills run

## Repository layout

```
cmd/                CLI commands (root, list, open, rm, serve, store, verify, extend,
                    verify-result, extend-start, extend-commit)
internal/config/    ~/.lathe/tutorials path resolution

internal/serve/     HTTP server, markdown renderer, embedded HTML templates, handoff endpoints
internal/extend/    NextPartFilename helper
.claude/skills/     lathe, lathe-verify, lathe-extend, lathe-ask skills (user-invoked, interactive)

```
